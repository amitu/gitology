I was working on a PHP application, and decided to use PEAR:DB after reading article at: http://www.onlamp.com/pub/a/php/2001/11/29/peardb.html

And things went fine till I was done developing and start testing the performance. The article said:
PEAR::DB certainly has problems and will continue to improve. Many critics of PEAR::DB, such as Manuel Lemos the author of Metabase, say PEAR::DB is too slow and doesn't provide a true abstraction to the interaction with database servers. I believe the most important part of a database abstraction library is its design -- I want to be able to use a high-quality library on my code; differences of 0.02% are not so important.

[Emphasis mine]

Where as my performance tests gave me:

script Throughput (Req.s/Sec.)
script1.php 42
script2.php 563

Where the script1.php is:
getVersion() {
return "PHP Soap Server 1.0";
}

include_once("DB.php ");
$server = new SoapServer("api.wsdl");
$server->addFunction("getVersion");
$server->handle();
?>

and script2.php has the "include_once db.php" line commented out.

This is an order of magnitude performance drop, not just 0.02 percent, DB.php, as nice it is, and I am sure its being worked on, but as it is, it should not be recommended for any serious application.
