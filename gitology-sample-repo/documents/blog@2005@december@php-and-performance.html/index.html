Sometime back I wrote in PHPs PEAR::DB Performance Sucks:

There is an order of magnitude performance drop, not just 0.02 percent, DB.php, as nice it is, and I am sure its being worked on, but as it is, it should not be recommended for any serious application.

This I later realized was not a good test, as someone commented out just now on the post. eAccelerator was the answer I was looking for, it increases performance of PHP scripts by caching them in compiled state, so that the overhead of compiling is almost completely eliminated. So yes the tests I did were accurate, and without caching, not only PEAR::DB, whole PHP's performance is pretty bad. On each request, the PHP script is looked up, parsed, and executed, and parsing is a major overhead if the size of script is significant. The problem is none of the tutorials or documentation I have read talk about this significant fact, maybe they just assume that everyone knows about it, or consider it as a part of PHP standard installtion. The performace boost is very significant, upto 400% in my installtion/application, and eAccelerator or any other script caching should be considered for all PHP installtions.
